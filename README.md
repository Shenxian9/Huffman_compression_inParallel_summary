# Huffman_compression_inParallel_summary

## 1. 项目简介

本项目实现了哈夫曼（Huffman）压缩算法的三个版本：

1. **串行（Serial）版本**
2. **OpenMP 并行（OpenMP）版本**
3. **CUDA 并行（CUDA）版本**

三个版本的操作逻辑完全一致，均支持对指定文件进行哈夫曼压缩（Compress）或解压缩（Decompress）。

---

## 2. 文件结构

假设当前目录为 `Huffman_compression_summary/`，其下主要文件/文件夹结构如下：

```
Huffman_compression_summary/
├── Chara.hpp           # 通用：字符处理相关头文件
├── Node.hpp            # 通用：哈夫曼树节点定义头文件
├── hfmTree.hpp         # 通用：哈夫曼树构建及操作头文件
│
├── main1.cpp           # 串行版本（Serial）源码
├── main2.cpp           # OpenMP 并行版本（OpenMP）源码
├── main3.cu            # CUDA 并行版本（CUDA）源码
│
└── (可选的测试输入输出文件夹，例如)
    ├── input/
    │   └── example.txt     # 示例输入文件
    └── output/
        └── example.hfm     # 示例压缩后输出（或解压后输出）
```

- **通用头文件（共用）**
  - `Chara.hpp`
  - `Node.hpp`
  - `hfmTree.hpp`
    这三份头文件在三个版本中都需要包含，定义了哈夫曼树节点、字符频率统计、哈夫曼树构建与遍历等基础功能。

- **串行版本源码**
  - `main1.cpp`
    - 实现哈夫曼压缩与解压缩的串行逻辑，直接使用 C++ 标准库完成所有操作。

- **OpenMP 并行版本源码**
  - `main2.cpp`
    - 基于 `main1.cpp`，在统计字符频率、构建哈夫曼树或编码/解码过程中适当插入 OpenMP 并行指令，以加速多核 CPU 上的计算。

- **CUDA 并行版本源码**
  - `main3.cu`
    - 使用 CUDA 在 GPU 上并行化计算，例如字符频率统计、优先队列并行化、编码/解码并行化等（具体实现请参见源码注释）。
    - 需要 NVIDIA GPU 及 CUDA Toolkit 环境（至少 CUDA 10.x 或更高版本）。

## 6. 实现细节与技术原理（重点：并行）

本节从**数据格式**、**串行基线**、**OpenMP 并行**、**CUDA 并行**四个角度说明当前实现的核心思路，方便你后续继续优化。

### 6.1 压缩文件格式与块化策略

三个版本共享同一套“头部 + 压缩块”格式，便于互相解压。

- 文件头（Header）大致包含：
  1. `blockCount`：块数量
  2. `totalChars`：原始字符总数
  3. `kinds`：字符种类数
  4. `kinds` 组 `(字符, 频次)`，用于重建哈夫曼树
- 每个压缩块（`PackedBlock`）包含：
  - `charCount`：该块对应的原文字符数
  - `bytes`：定长 bit 打包区（`DATA_BYTES`）

当前代码将块大小定义为：

- `BLOCKSIZE = 8 * 8 * 1024 bits`
- 其中 32 bit 预留给 `charCount`
- 有效编码空间 `DATA_BITS = BLOCKSIZE - 32`

这种设计的好处是：

- 解压时每块可独立恢复（天然适合并行）
- 头部频次信息可跨平台重建同一棵哈夫曼树

### 6.2 串行版本（main1.cpp）流程

串行实现是其余并行版的语义基线，流程如下：

1. 读入完整输入文件到内存
2. 统计 128 个 ASCII 字符频次
3. 构建哈夫曼树并生成编码表
4. 逐字符查表编码并写入块缓冲
5. 写文件头 + 块数据
6. 解压时反向读取：重建树、逐块解码、顺序回写

特点：

- 逻辑最直接、易调试
- 吞吐受单核 CPU 限制
- 对大文件而言，频次统计和编码环节耗时占比高

### 6.3 OpenMP 并行实现（main2.cpp）

OpenMP 版本的核心思想是：**保持文件格式不变，仅并行化耗时阶段**。

#### 6.3.1 并行频次统计（Compression）

- 使用 `#pragma omp parallel` 创建线程组。
- 每个线程对自己分配到的 `[lo, hi)` 子区间做本地频次统计（`local[128]`）。
- 再通过 `#pragma omp critical` 把本地桶累加到全局 `freq`。

优点：

- 避免每个字符都做全局原子更新（开销大）
- 仅在最终归并时进入临界区，冲突显著减少

#### 6.3.2 并行分段压缩（Compression）

- 各线程分别处理自己字符区间，独立维护 `PackedBlock curr`。
- 当当前字符编码放不进块时，先封块（写 `charCount`）再继续。
- 每线程得到 `localBlocks`，最后主线程按线程号顺序拼接为 `blocks`。

这属于“线程私有构建 + 最终拼接”模式，能有效减少锁竞争。

#### 6.3.3 并行逐块解码（Decompression）

- 先把所有块读入内存。
- 用 `#pragma omp parallel for schedule(dynamic)` 并行处理每个块。
- 每个线程独立把字节展开为 bit 串，再按哈夫曼树恢复字符，直到达到该块 `charCount`。

为什么并行效果通常不错：

- 块之间没有数据依赖
- 每块的输出缓冲独立，几乎无写冲突

#### 6.3.4 OpenMP 版本可能的性能瓶颈

- `critical` 归并频次在超多线程下仍可能成为热点。
- 使用 `string` 存储 bit 串会有额外内存分配与拷贝开销。
- 分线程按字符区间切分时，若编码长度分布不均，可能出现负载不平衡。

### 6.4 CUDA 并行实现（main3.cu）

CUDA 版本采用“**CPU 构树 + GPU 并行统计/编码/解码**”的混合方案。

#### 6.4.1 GPU 频次统计

- `countFreqKernel`：每个线程处理一个输入字符，使用 `atomicAdd` 更新设备端频次数组。
- 优势：充分利用 GPU 大规模并行线程。
- 代价：高频字符会造成原子冲突（热点桶竞争）。

#### 6.4.2 编码表组织与传输

- 哈夫曼树与编码表先在 CPU 侧构建。
- 将每个字符编码串复制到 GPU，并构造 `d_codeTable`（指针表）+ `d_codeLens`（长度表）。
- GPU 内核根据字符索引直接查指针和长度执行编码。

这种“主机端构树、设备端查表”的方式实现成本较低，但会引入较多小块内存分配/拷贝。

#### 6.4.3 块划分与编码内核

- 主机端先根据字符编码长度预估，得到 `blockStarts`（每块字符起止）。
- `encodeBlocksKernel` 以“1 block 对应 1 逻辑压缩块”的方式执行。
- 内核中使用共享内存缓存块内 bit 打包结果，再回写到全局输出。

当前实现特点：

- 编码 kernel 启动参数为 `<<<numBlocks, 1, DATA_BYTES>>>`（每个压缩块仅用 1 个线程执行核心逻辑）。
- 这样实现简单，但并未充分利用 block 内并行度，属于“可运行但仍有较大优化空间”的设计。

#### 6.4.4 GPU 解码内核

- 将压缩字节区、每块字符数、前缀输出偏移（`prefixCounts`）拷入设备。
- `decodeBlocksKernel` 每个 CUDA block 负责解一个压缩块。
- 通过前缀和定位输出写入起点，避免块间输出覆盖。

注意：哈夫曼解码本身具有“按位逐步匹配”的顺序依赖，天然不如统计类任务易并行。

#### 6.4.5 CUDA 版本的关键权衡

- 优点：
  - 在大文件场景，频次统计和块级并行可获得加速。
- 局限：
  - 主机/设备数据传输开销明显。
  - 当前编码 kernel 的线程利用率偏低（每块主要由 thread 0 工作）。
  - 解码阶段仍有较强串行匹配特性。

### 6.5 三个版本并行策略对比

| 维度       | 串行         | OpenMP                     | CUDA                             |
| ---------- | ------------ | -------------------------- | -------------------------------- |
| 频次统计   | 单线程       | 线程私有桶 + critical 归并 | 大量线程 + 原子加                |
| 编码       | 单线程逐字符 | 按区间并行，线程私有块     | 块级并行，当前实现块内并行度较低 |
| 解码       | 单线程逐块   | 块级 `parallel for`        | 块级 kernel + 前缀偏移           |
| 开发复杂度 | 低           | 中                         | 高                               |
| 典型适用   | 小文件/调试  | 多核 CPU 通用场景          | 大规模数据 + 可接受传输开销      |

### 6.6 后续可优化方向（并行重点）

1. **OpenMP 频次归并优化**
   - 用二维频次数组（`thread_count x 128`）+ 末端串行归并，减少 `critical` 开销。
2. **OpenMP 编码负载均衡**
   - 采用按“估计 bit 数”切分任务，而非仅按字符数均分。
3. **CUDA 编码内核并行化**
   - 让一个块内多个线程协作打包，减少 thread 0 串行负担。
4. **CUDA 内存管理优化**
   - 将编码表压成连续数组 + 偏移表，减少大量 `cudaMalloc` 小对象。
5. **解码结构优化**
   - 可尝试构建有限状态机（FSM）或分层查表，提高按位匹配效率。
