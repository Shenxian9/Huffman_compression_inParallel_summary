# Huffman Compression in Parallel: Serial / OpenMP / CUDA

一个围绕 **哈夫曼压缩（Huffman Coding）** 的并行实现练习项目，包含三套可运行版本：

- **Serial**：串行基线实现
- **OpenMP**：面向多核 CPU 的并行实现
- **CUDA**：面向 NVIDIA GPU 的并行实现

项目目标围绕同一套哈夫曼压缩流程，比较 **串行、CPU 多线程、GPU 并行** 三种实现路径在实际文件处理中的性能差异，并分析它们各自的适用场景。

---

## 1. 项目概览

### 1.1 主要功能

三个版本均支持：

- 对指定文件进行 **压缩（Compress）**
- 对 `.hfm` 压缩文件进行 **解压（Decompress）**
- 使用统一的压缩文件格式，保证不同版本之间可互相解压

### 1.2 项目特点

- 以 **串行版本** 作为语义基线，便于验证正确性
- 以 **块压缩（block-based compression）** 为核心组织方式，提升并行友好性
- OpenMP 版本重点并行化 **频次统计、分段编码、逐块解码**
- CUDA 版本采用 **CPU 构树 + GPU 并行统计/编码/解码** 的混合方案
- 提供统一控制程序，可在三种实现之间快速切换

### 1.3 适用场景

- 想学习哈夫曼压缩的基本实现流程
- 想对比 OpenMP 与 CUDA 的并行编程思路
- 想做并行计算课程、大作业或性能分析类展示
- 想在已有压缩算法基础上继续做优化实验

## 3. 设计思路

### 3.1 为什么要做并行化

哈夫曼编码本身是经典无损压缩算法，但传统串行实现面对大文件时，通常会在以下阶段出现明显瓶颈：

- 输入扫描与字符频次统计
- 逐字符编码与 bit 打包
- 解压时按位匹配哈夫曼树

本项目的核心思路是：

1. 保持压缩 / 解压逻辑一致；
2. 先实现可靠的串行基线；
3. 再逐步将高耗时阶段映射到多核 CPU 与 GPU；
4. 最终比较不同并行策略的收益与代价。

### 3.2 块压缩策略

为了提高并行友好性，项目采用了 **块压缩** 的组织方式。压缩文件由：

- **文件头（Header）**
- **若干压缩块（PackedBlock）**

两部分组成。

文件头中包含：

- `blockCount`：块数
- `totalChars`：原始字符总数
- `kinds`：字符种类数
- 若干组 `(字符, 频次)`：用于解压时重建哈夫曼树

每个压缩块中包含：

- `charCount`：该块对应的原始字符数量
- `bytes`：定长 bit 打包区

当前实现中：

- `BLOCKSIZE = 8 * 8 * 1024 bits`
- 其中 32 bit 预留给 `charCount`
- 有效数据区为 `DATA_BITS = BLOCKSIZE - 32`

这样设计的直接好处是：

- 每个块都可以独立解压，天然适合并行；
- 三个版本共享同一文件格式，便于交叉验证正确性；
- 块级别组织更适合后续继续做流水线、异构调度等优化。

## 4. 三个版本的实现差异

## 4.1 Serial：串行版本

串行版本是整个项目的功能基线，主要流程如下：

1. 读入完整输入文件
2. 统计 128 个 ASCII 字符频次
3. 构建哈夫曼树并生成编码表
4. 逐字符查表并写入压缩块
5. 输出文件头与块数据
6. 解压时重建树、逐块恢复并回写文件

特点：

- 逻辑清晰，最容易调试
- 适合验证功能正确性
- 对大文件处理时性能受单核限制明显

## 4.2 OpenMP：多核 CPU 并行版本

OpenMP 版本尽量保持文件格式和总体逻辑不变，只对耗时阶段并行化。

### 并行频次统计

- 使用 `#pragma omp parallel` 建立线程组
- 每个线程在本地区间内维护 `local[128]`
- 最后将各线程的局部统计结果归并到全局频次数组

### 并行分段压缩

- 不同线程处理不同字符区间
- 每个线程维护自己的 `localBlocks`
- 主线程最终按线程顺序拼接结果

### 并行逐块解压

- 将所有压缩块读入内存
- 使用 `#pragma omp parallel for` 对块做并行解码
- 各块恢复完成后按原顺序写回输出文件

这一版本的优点是：

- 对现有串行代码改动相对有限
- 在多核 CPU 上能较稳定获得加速
- 启动成本低，适合中小规模数据

## 4.3 CUDA：GPU 并行版本

CUDA 版本采用 **CPU 构树 + GPU 并行计算** 的混合架构。

### GPU 频次统计

- `countFreqKernel` 中每个线程处理一个输入字符
- 通过 `atomicAdd` 更新设备端频次数组

### GPU 编码

- 编码表先在 CPU 侧构建
- 再复制到 GPU 端供 kernel 查表使用
- 主机端预估每个压缩块的字符范围，随后由编码 kernel 处理

### GPU 解码

- 每个 CUDA block 负责一个逻辑压缩块
- 结合每块字符数和前缀偏移量进行并行恢复

这一版本的特点是：

- 大文件场景下更有机会发挥硬件吞吐优势
- 实现复杂度和调试成本显著高于前两者
- 主机 / 设备数据传输、原子冲突、块内线程利用率都是需要继续优化的点

## 5. 环境要求

### 5.1 硬件

- Serial / OpenMP：普通 x86_64 CPU 即可
- CUDA：需要 NVIDIA GPU 及可用驱动

### 5.2 软件

- `g++`（建议支持 C++11 及以上）
- OpenMP（编译时开启 `-fopenmp`）
- CUDA Toolkit（使用 `nvcc` 编译 CUDA 版本）

## 8. 实验与性能结论

根据配套实验报告，测试数据集使用 **1MB–1024MB 的随机文本文件**，主要考察：

- 压缩时间
- 解压时间
- 压缩比
- 加速比
- 吞吐率

这些指标与实验方案在报告中已有明确说明。fileciteturn1file5

### 8.1 结论概括

报告给出的总体结论可以概括为：

- 与串行版本相比，并行化显著降低了压缩和解压耗时。
- OpenMP 在较小规模数据下就能较快发挥性能，且启动开销较低。
- CUDA 的优势更多体现在中大规模数据上；随着数据规模增大，GPU 并行能力会更充分地释
- 三种实现的压缩比整体接近，说明并行化优化主要改善的是执行效率，而不是改变压缩语义。报告将压缩比概括为约 `82.5%`（因为样本为随机生成的ASCII字符，符合期望值E(压缩后码长)=$log_2{128}$=7,即<7/8=87.5%)。

### 8.2 如何理解这组结果

- **小规模数据**：OpenMP 往往更划算，因为无需承担 GPU 数据传输和 kernel 启动开销。
- **中大规模数据**：CUDA 更有潜力取得更高吞吐。
- **课程展示或工程对比**：三版本并存非常适合做“相同算法在不同并行模型下的实现比较”。

## 9. 当前实现的局限

虽然三个版本都能运行并完成互相兼容的压缩 / 解压，但当前实现仍保留了若干明显的优化空间：

- OpenMP 频次归并阶段仍可能受 `critical` 影响
- OpenMP 分段压缩按字符数量划分，可能存在负载不均
- CUDA 编码 kernel 当前块内并行度偏低
- 主机 / 设备之间存在额外内存分配与拷贝开销
- 哈夫曼解码本身带有按位匹配的顺序依赖，不如统计阶段那样容易并行化

这些问题在原 README 和实验报告中都已被提到。fileciteturn1file9

## 10. 后续可改进方向

结合现有 README 与实验报告，后续优化可以优先考虑：

1. **OpenMP 频次统计去 `critical`**  
   使用二维局部频次数组统一归并，降低热点同步开销。

2. **OpenMP 负载均衡优化**  
   从“按字符数均分”改为“按估计 bit 数均分”。

3. **CUDA 编码 kernel 深化并行化**  
   让一个 block 内多个线程协同打包，而不只是由单线程承担主要工作。

4. **编码表内存布局优化**  
   用连续数组 + 偏移表替代大量小对象分配，降低 CUDA 侧管理开销。

5. **流水线并行与异构协同**  
   将文件读写、构树、编码、解码进一步拆分，实现 I/O 与计算重叠；必要时可扩展到多 GPU 或混合 OpenMP + CUDA 流水线方案。报告结论部分也给出了这一方向。fileciteturn1file2
